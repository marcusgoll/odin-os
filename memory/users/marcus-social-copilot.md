# Marcus Social Copilot Memory

## Voice And Style Preferences

- sound professional, clear, and useful
- write at peer level, not as a lecturer talking down to students
- avoid fake hype, fake hustle, generic inspiration, and empty motivational framing
- prefer sharp hooks, practical lessons, and operational clarity
- on X, lean toward Marcus's inner thoughts, perspective, conviction, tension, and concise observations
- on LinkedIn, use a more professional framing with clearer structure, practical lessons, and peer-level authority

## Topic Pillars

- aviation training and flight instruction
- student pilot mistakes and teaching patterns
- airline career progression
- pilot professionalism and judgment
- practical lessons Marcus has seen in real instruction

## Approval And Compliance

- every post requires Marcus approval before publishing
- every reply requires Marcus approval before posting
- sensitive topics need explicit review and may be a no-reply
- no stealth automation, no fake personas, no evasive browser automation, and no platform-violating engagement behavior
- for live browser evidence, start with read-only visible-page capture from explicit X post URLs
- when reviewing a full week of X outcomes, prefer the weekly explicit-post bundle so each observed post still lands as its own `social_evidence` record
- keep LinkedIn evidence manual until there is a compliant, reviewable path inside Odin

## Repo Authority

- for Odin implementation work supporting this social copilot, treat `odin-os` as the canonical runtime root
- `odin-orchestrator` is migration context only and is being phased out

## Content History Rules

- keep approved post summaries
- keep rejected draft lessons when they reveal a durable style preference
- avoid storing unrelated personal or sensitive memory in this social context

## Performance Learning Rules

- remember which hooks, topics, and structures Marcus likes approving
- remember which formats perform well with the right audience
- do not overfit to vanity metrics or one-off spikes
- when building retrospectives, keep the last 7 days primary and compare them against the last 4 rolling weekly windows for recurring versus new signals
- use those recurring and new signals to generate prompt-only next-week keep, avoid, and test guidance before adding any new persistence
- when X visible evidence exists, include it in the weekly retrospective as operator-observed page evidence rather than implied platform telemetry
